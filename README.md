- Improve existing features without adding entirely new functionality
- Example: "Improve search UX with better filtering"
- Phases: UX Analysis → A/B Testing → Implementation → Validation

- Code quality improvements with **ZERO behavior changes**
- Critical Principle: If tests need modification, it's NOT refactoring
- Example: "Extract duplicated validation logic to shared utility"
- Phases: Test coverage → Refactor → Verify (tests unchanged)

**6. performance-optimization.md** (MINOR or PATCH versions)
- Speed, scalability, resource usage improvements
- Requires quantifiable metrics (e.g., "50%+ improvement")
- Example: "Optimize database queries for dashboard (3.2s → 1.8s = 44% improvement)"
- Phases: Baseline → Profile → Optimize → Verify improvement

**7. security-patch.md** (PATCH versions)
- CVE fixes, security vulnerability remediation
- Priority: HIGH (CRITICAL issues use hotfix.md)
- Example: "Fix SQL injection vulnerability in user search"
- Phases: CVE analysis → Patch → Security testing → Disclosure

**8. version-upgrade.md** (PATCH/MINOR/MAJOR versions)
- Framework, language, or dependency major version upgrades
- Example: "Upgrade Next.js 14 → 15 (App Router migration)"
- Phases: Assessment → Research → Migration → Extensive testing

Every release goes through **staged rollout** with monitoring and rollback capability.

**5-Phase Release Process**:

**Phase 1: Pre-Release Preparation**
- Final quality gates validation (all 4 gates MUST PASS)
- Release notes generation (from `.memory/version-history.md`)
- Stakeholder notification (optional)
- Rollback plan confirmation (5-10 minute rollback time)

**Phase 2: Canary Deployment** (10% traffic)
- Deploy to 10% of users
- Monitor for 2-4 hours
- Check: Error rates (<1% target), performance metrics (no regression), user feedback
- **Decision**: Proceed or rollback

**Phase 3: Gradual Rollout** (50% traffic)
- Expand to 50% of users
- Monitor for 4-8 hours
- Validate success metrics
- **Decision**: Proceed or rollback

**Phase 4: Full Deployment** (100% traffic)
- Complete rollout to all users
- Monitor for 24 hours
- Collect user feedback (`.memory/user-feedback.md`)

**Phase 5: Post-Release Validation**
- Track production metrics (`.memory/production-metrics.md`)
- Monitor error rates, performance, user feedback
- Respond to incidents (route to hotfix.md if CRITICAL)
- Document lessons learned
- Return to 09-continuous-development.md for next work item ♾️

**Rollback Strategy**:
- Automated rollback on health check failure
- Manual rollback via pipeline trigger
- Rollback time: 5-10 minutes
- Zero-downtime rollback (blue-green deployment)

**Success Criteria**:
- Error rate <1% (vs. pre-release baseline)
- Performance metrics stable (no regressions)
- User feedback positive (NPS maintained or improved)
- No critical incidents within 48 hours

**Lifecycle-Related Questions**:

**Q: What happens after v1.0.0 is deployed?**
A: The system automatically transitions to **continuous_development** state and enters an infinite improvement loop. It monitors production metrics, user feedback, and incidents, then autonomously develops features, fixes bugs, optimizes performance, and releases updates (v1.1.0, v1.1.1, etc.). You can request new features anytime, or the system responds to production issues automatically.

**Q: How does the system decide between MAJOR, MINOR, and PATCH versions?**
A: Semantic versioning is applied automatically:
- **MAJOR (v1.x.x → v2.0.0)**: Breaking changes, API incompatibilities, fundamental architecture changes
- **MINOR (v1.2.x → v1.3.0)**: New features, backward-compatible additions
- **PATCH (v1.2.3 → v1.2.4)**: Bug fixes, performance improvements, security patches, refactoring

pm-orchestrator analyzes the work type and impact, then decides version bump autonomously.

**Q: Can I pause continuous development and just keep v1.0.0 running?**
A: Yes! Continuous development only activates when you request changes or when production issues are detected. If you don't request anything and there are no critical production issues, the system simply monitors and doesn't make changes. However, security patches and critical hotfixes will still be recommended.

**Q: What's the difference between bug-fix and hotfix workflows?**
A:
- **bug-fix.md**: Regular bugs (non-critical), standard pipeline, typical resolution time: 5-15 minutes
- **hotfix.md**: CRITICAL production issues (service down, data loss risk), EXPEDITED pipeline, target MTTR: 15 minutes, goes through rapid deployment

**Q: How does Deep Thinking Protocol affect development time?**
A: Deep Thinking adds 2-5 minutes upfront for complex decisions, BUT reduces overall time-to-production by 15-20% because:
- Fewer revisions (decision revision rate <10% vs 30-40%)
- Higher first-time-right rate (>70% vs 40-50%)
- Fewer post-deployment bugs (-40-60%)

The extra thinking time is more than recovered by avoiding rework.

**Q: Can I see the Deep Thinking analysis process?**
A: Yes! All Category A (MANDATORY) decisions are documented in `.memory/decisions.md` with:
- Problem statement
- Alternatives considered (3-5 options)
- Evaluation matrix (across 6 dimensions)
- Decision rationale
- Tradeoffs and risks
- Implementation plan

**Q: What if I want to skip a PATCH version and go straight to MINOR?**
A: You can request it explicitly: "Add feature X and skip to v1.3.0". The system will accumulate multiple changes and release them together as MINOR version. However, critical security patches and hotfixes will still be released immediately as PATCH versions.

**Q: How do I know what's in the next release?**
A: Check `.memory/release-plan.md` - it contains:
- Next release version and type (MAJOR/MINOR/PATCH)
- Planned features and fixes
- Release timeline
- Backlog items
- Release criteria

**Q: Can the system handle multiple features in parallel?**
A: Currently, the system processes work items sequentially (one sub-workflow at a time) to maintain quality and avoid conflicts. However, planning for concurrent development is in the roadmap (v1.1).

**Q: What happens if a staged rollout fails?**
A: Automatic rollback:
1. Health checks detect failures (error rate spike, performance degradation)
2. System automatically rolls back to previous version (5-10 minute rollback time)
3. Incident logged in `.memory/incident-log.md`
4. Issue routed to appropriate sub-workflow (hotfix.md if CRITICAL, bug-fix.md if not)
5. Root cause analysis performed
6. Fix developed and re-deployed
